# Design, Prototyping, and Implementation

## Specifications and Design

- Specifications come after requirements and describe how the system will be built. [L07 p1](<../Lecture Slides/07 - Specifications.pdf#page=1>) [L08 p4](<../Lecture Slides/08 - Prototyping.pdf#page=4>)
- Good teams plan their approach, agree what good work looks like, review work together, and avoid simply splitting work up without integration. [L07 p8](<../Lecture Slides/07 - Specifications.pdf#page=8>)
- Low-level designs guide developers and are final outputs of the specification phase. [L09 p7](<../Lecture Slides/09 - Object-Oriented Design and Test Planning.pdf#page=7>)
- Test planning is part of the specification phase because expected behaviour must be testable before implementation. [L09 p7](<../Lecture Slides/09 - Object-Oriented Design and Test Planning.pdf#page=7>)

## Prototyping

- Prototypes are high-level designs that help develop specifications. [L08 p7](<../Lecture Slides/08 - Prototyping.pdf#page=7>)
- Different prototype types test different ideas. [L08 p7](<../Lecture Slides/08 - Prototyping.pdf#page=7>)
- Prototypes are useful for communication and checking design ideas with stakeholders. [L08 p7](<../Lecture Slides/08 - Prototyping.pdf#page=7>)
- Prototyping carries risks, so prototype scope and purpose should be clear. [L08 p7](<../Lecture Slides/08 - Prototyping.pdf#page=7>)
- Low-fidelity prototypes are rough and cheap, while high-fidelity prototypes are more realistic and detailed. [L08 p7](<../Lecture Slides/08 - Prototyping.pdf#page=7>)

## Implementation Quality

- Software engineering development is social: developers code for, alongside, and sometimes with other people. [L10 p8](<../Lecture Slides/10 - Implementation.pdf#page=8>)
- Well-implemented code is easy to maintain. [L10 p8](<../Lecture Slides/10 - Implementation.pdf#page=8>)
- Implementation quality depends on coding standards, conventions, and strategies. [L10 p8](<../Lecture Slides/10 - Implementation.pdf#page=8>)
- Debugging is a social and technical skill, especially when working with code written by other people. [L10 p8](<../Lecture Slides/10 - Implementation.pdf#page=8>) [L11 p7](<../Lecture Slides/11 - Unit Tests and TDD.pdf#page=7>)
- Unit testing aims to establish that bugs exist; debugging aims to locate and remove their causes. [L11 p8](<../Lecture Slides/11 - Unit Tests and TDD.pdf#page=8>)
- Debugging strategies include breakpoints, stepping, variable inspection, trace tables, print/log statements, hypothesis testing, and systematic exclusion of causes. [L11 p11](<../Lecture Slides/11 - Unit Tests and TDD.pdf#page=11>) [L11 p12](<../Lecture Slides/11 - Unit Tests and TDD.pdf#page=12>) [L11 p13](<../Lecture Slides/11 - Unit Tests and TDD.pdf#page=13>) [L11 p14](<../Lecture Slides/11 - Unit Tests and TDD.pdf#page=14>)
- Randomly changing code is bad debugging practice because it can hide causes or create new faults. [L11 p15](<../Lecture Slides/11 - Unit Tests and TDD.pdf#page=15>)

## Traceability and Coursework-Style Evidence

- Good engineering evidence connects issues, markdown documentation, code comments, and tests. [L15 p14](<../Lecture Slides/15 - Coursework Intro and Evolution Maintenance.pdf#page=14>)
- Traceability matters because test plans should connect back to requirements, issues, and implementation evidence. [L15 p15](<../Lecture Slides/15 - Coursework Intro and Evolution Maintenance.pdf#page=15>)
- Good practice includes issues, branching, merge requests, templates, code reviews, coding conventions, and naming conventions. [L15 p12](<../Lecture Slides/15 - Coursework Intro and Evolution Maintenance.pdf#page=12>)

## Exam Angles

- If asked why coding conventions help quality, say they improve readability, maintainability, consistency, review quality, and team integration. [L10 p8](<../Lecture Slides/10 - Implementation.pdf#page=8>) [L15 p12](<../Lecture Slides/15 - Coursework Intro and Evolution Maintenance.pdf#page=12>)
- If asked about pair programming, connect it to social coding, continuous review, knowledge sharing, and reducing dependence on one person. [L10 p8](<../Lecture Slides/10 - Implementation.pdf#page=8>) [L16 p68](<../Lecture Slides/16 - Agile vs Traditional and Maintenance.pdf#page=68>)
- If asked about prototyping, explain its role in developing specifications, checking ideas, communicating with stakeholders, and managing uncertainty. [L08 p7](<../Lecture Slides/08 - Prototyping.pdf#page=7>)
