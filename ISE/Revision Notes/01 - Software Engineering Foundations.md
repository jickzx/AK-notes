# Software Engineering Foundations

## Why Software Engineering Exists

- Real software is more than code: it can include compiled code, documentation, configuration, installation, upgrade, deployment, servers, databases, and platform versions. [L01 p17](<../Lecture Slides/01 - Introduction to Software Engineering.pdf#page=17>) [L01 p21](<../Lecture Slides/01 - Introduction to Software Engineering.pdf#page=21>) [L01 p26](<../Lecture Slides/01 - Introduction to Software Engineering.pdf#page=26>)
- Software engineering applies a systematic, disciplined, organised approach to developing, operating, and maintaining software. [L01 p27](<../Lecture Slides/01 - Introduction to Software Engineering.pdf#page=27>) [L01 p56](<../Lecture Slides/01 - Introduction to Software Engineering.pdf#page=56>)
- Software engineering is needed because individual coding approaches do not scale well to large systems, real users, multiple developers, changing versions, and long-term maintenance. [L01 p15](<../Lecture Slides/01 - Introduction to Software Engineering.pdf#page=15>) [L01 p22](<../Lecture Slides/01 - Introduction to Software Engineering.pdf#page=22>) [L01 p34](<../Lecture Slides/01 - Introduction to Software Engineering.pdf#page=34>)
- Common project failure causes include poor requirements, communication breakdown, insufficient testing, cost overruns, delays, and cancelled projects. [L01 p37](<../Lecture Slides/01 - Introduction to Software Engineering.pdf#page=37>) [L03 p13](<../Lecture Slides/03 - Requirements Introduction.pdf#page=13>)
- Cutting corners to reduce cost or hurry a project usually lowers quality or increases failure risk. [L01 p47](<../Lecture Slides/01 - Introduction to Software Engineering.pdf#page=47>)

## Core Software Engineering Activities

- Specification defines what the system should do and the constraints on development. [L01 p58](<../Lecture Slides/01 - Introduction to Software Engineering.pdf#page=58>)
- Development is the production of the software system. [L01 p58](<../Lecture Slides/01 - Introduction to Software Engineering.pdf#page=58>)
- Validation checks whether the software is what the customer wants. [L01 p58](<../Lecture Slides/01 - Introduction to Software Engineering.pdf#page=58>) [L03 p4](<../Lecture Slides/03 - Requirements Introduction.pdf#page=4>)
- Evolution changes code in response to new requirements or demands. [L01 p58](<../Lecture Slides/01 - Introduction to Software Engineering.pdf#page=58>) [L03 p4](<../Lecture Slides/03 - Requirements Introduction.pdf#page=4>)
- These activities appear in different processes even when the process model changes. [L01 p57](<../Lecture Slides/01 - Introduction to Software Engineering.pdf#page=57>) [L01 p58](<../Lecture Slides/01 - Introduction to Software Engineering.pdf#page=58>)

## Process Models

- The waterfall model moves through requirements definition, system/software design, implementation and unit testing, integration/system testing, then operation and maintenance. [L01 p59](<../Lecture Slides/01 - Introduction to Software Engineering.pdf#page=59>)
- Waterfall and V-model approaches need stable requirements and can be inflexible when changes have knock-on effects. [L01 p61](<../Lecture Slides/01 - Introduction to Software Engineering.pdf#page=61>)
- Iterative/agile models allow specification, development, and validation to overlap through intermediate versions. [L01 p62](<../Lecture Slides/01 - Introduction to Software Engineering.pdf#page=62>)
- The spiral model repeatedly combines risk analysis, prototyping, requirements, validation, design, testing, review, and planning the next phase. [L01 p63](<../Lecture Slides/01 - Introduction to Software Engineering.pdf#page=63>) [L16 p29](<../Lecture Slides/16 - Agile vs Traditional and Maintenance.pdf#page=29>)
- Agile models can suffer from poor process visibility, weak structure, too much doing without planning, and lightweight documentation being mistaken for no documentation. [L01 p64](<../Lecture Slides/01 - Introduction to Software Engineering.pdf#page=64>)
- Maintaining and evolving software often costs more than producing it in the first place. [L01 p65](<../Lecture Slides/01 - Introduction to Software Engineering.pdf#page=65>)

## Exam Angles

- If asked to explain why software engineering matters, mention scale, teams, reliability, cost, schedule, quality, and maintenance.
